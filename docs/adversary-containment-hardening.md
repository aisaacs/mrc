## Why

The adversary feature (`summon_adversary` ‚Üí "Pierre") + the rooms daemon were modeled as a **network cage** ("no internet = contained"). A live red-team ‚Äî the *summoned adversary itself*, post-rebuild, from inside its own cage ‚Äî proved the model is broadly holey: **the cage stops packets and ignores `cat`**, and **the daemon authenticates nothing it consumes.** 8 grounded findings, 2 refuted, 1 cleared. (The four substrate checks DID pass ‚Äî belt 2 blocks arbitrary 443, the env bits are set, the model endpoint is reachable ‚Äî so the network cage works *against the threat it was built for*; these findings are the threats it was NOT built for.)

## The two spines (every fix flows from one of these)

**Spine 1 ‚Äî authenticate everything the daemon consumes.** Today the daemon trusts the wire for every trust signal: register identity (G), the peer-text trust-prefix (E), the dashboard origin (H). All forgeable. Make each verified or structural.

**Spine 2 ‚Äî fail closed on absence, done right.** #32's own invariant (always-write ‚Üí absence anomalous ‚Üí fail closed) ‚Äî which both enforcers currently *violate* (they fail OPEN on absence):
- **Firewall (C):** stays 2-state, BUT the profile SOURCE must flip from the caller-settable env `${MRC_ADVERSARY_FW:-}` (absent‚Üínormal = today's fail-OPEN) to an **always-written root-owned file** (=0 normal / =1 adversary), **absent‚ÜíHARDEN**. 2-state is safe here *because over-caging fails SAFE* ‚Äî a wrongly-hardened normal session loses npm/openai/sentry but still reaches the model; more-restrictive is always the safe direction. Plus a **LOUD boot-log warning** on the harden-on-absent branch (diagnosability). ‚öÝÔ∏è Guard in review: do NOT let a cleanup revert the source to the env check ‚Äî same code, opposite default, re-opens the hole.
- **Daemon (B):** 3-STATE via `classifySession` (the helper already built for #32's resume guard, `src/session-record.js`), NOT 2-state `isAdversarySession`. record=adversary‚Üícage+flag; record=normal‚Üítrust; record=ABSENT/unreadable‚Üí**LOUD fail-closed**: refuse to seat as a trusted-normal peer AND do not silently brand adversary; surface `[unverifiable session ‚Äî security record missing]` to the human. 3-state (not the firewall's 2-state) because the daemon's mislabel has teeth ‚Äî silently branding a legit absent-record session (pre-#32 legacy, a human-wiped record, an un-run backfill) as adversary is a *new* availability bug.

## Live test + A/#40 refutation ‚Äî and the option-1 proxy plan (NEXT BUILD)

**A/#40 was refuted in the live test**, with a hard receipt. A caged Pierre ran
`curl -k --resolve example.com:443:160.79.104.10 https://example.com/` ‚Üí a **full TLS handshake completed** to the pinned Anthropic IP, and the edge served a cert for **CN=example.com** (`server: cloudflare`, a `cf-ray`). Controls: a foreign SNI at a NON-pinned IP (8.8.8.8) died at L3/L4 (`rc=7` ‚Üí the firewall blocks by IP, and there is NO local TLS-inspecting proxy); a second foreign host at the pinned IP rode identically. So `160.79.104.10` (api.anthropic.com) is a **shared Cloudflare edge** ‚Äî the cage pins the IP but not the SNI, the iptables/ipset firewall is L7-blind, and the old boot-check was a false-green (a 4xx with `rc=0` is the foreign origin ANSWERING, not a refusal ‚Äî the same probe-1 trap, re-shipped). **The cage is network-restricted but NOT exfil-sealed:** a malicious/injected adversary can `--resolve attacker-cf-zone:443:<pinned-ip>` and POST the repo out to an attacker-owned Cloudflare zone.

**Option 2 (LANDED this commit):** the boot probe is now diagnostic-only + rc-based ‚Äî it logs a LOUD "cage is NOT exfil-sealed" WARNING (does NOT abort), and the docs + the launch banner stop claiming a seal. Feature stays usable; the hole is documented, not hidden.

**Option 1 (THE NEXT BUILD ‚Äî the real seal): a host-side SNI-pinning egress proxy.**
- Cage gets **zero direct 443**: for the adversary profile, drop the ipset/443 OUTPUT allow and allow ONLY the new proxy port; set `HTTPS_PROXY`/`HTTP_PROXY` in the adversary env so Claude Code routes through it.
- The proxy (a new host process beside clipboard/notify, on `host.docker.internal:<port>`) accepts `CONNECT host:443`, then **peeks the ClientHello SNI inside the tunnel** and forwards only if SNI ‚àà {`api.anthropic.com`, `platform.claude.com`}, dropping everything else. Peek-and-tunnel ‚Äî **NO MITM/cert** needed (the ClientHello SNI is exactly what the CF edge routes on, so allowlisting it pins the real destination). ‚öÝ Checking only the CONNECT *host* is INSUFFICIENT ‚Äî an adversary can `CONNECT api.anthropic.com` but ride a foreign SNI *inside* the tunnel; the proxy MUST inspect the in-tunnel SNI.
- **Verify (the gate):** re-run the `--resolve` ride against it ‚Äî it must die at the connection layer (rc 7/28/35), NOT return a 403. Confirm a real Anthropic call still works (model + statusline rate gauges). Confirm NORMAL sessions are untouched ‚Äî the proxy is adversary-cage-only.
