## Why

The adversary feature (`summon_adversary` → "Pierre") + the rooms daemon were modeled as a **network cage** ("no internet = contained"). A live red-team — the *summoned adversary itself*, post-rebuild, from inside its own cage — proved the model is broadly holey: **the cage stops packets and ignores `cat`**, and **the daemon authenticates nothing it consumes.** 8 grounded findings, 2 refuted, 1 cleared. (The four substrate checks DID pass — belt 2 blocks arbitrary 443, the env bits are set, the model endpoint is reachable — so the network cage works *against the threat it was built for*; these findings are the threats it was NOT built for.)

## The two spines (every fix flows from one of these)

**Spine 1 — authenticate everything the daemon consumes.** Today the daemon trusts the wire for every trust signal: register identity (G), the peer-text trust-prefix (E), the dashboard origin (H). All forgeable. Make each verified or structural.

**Spine 2 — fail closed on absence, done right.** #32's own invariant (always-write → absence anomalous → fail closed) — which both enforcers currently *violate* (they fail OPEN on absence):
- **Firewall (C):** stays 2-state, BUT the profile SOURCE must flip from the caller-settable env `${MRC_ADVERSARY_FW:-}` (absent→normal = today's fail-OPEN) to an **always-written root-owned file** (=0 normal / =1 adversary), **absent→HARDEN**. 2-state is safe here *because over-caging fails SAFE* — a wrongly-hardened normal session loses npm/openai/sentry but still reaches the model; more-restrictive is always the safe direction. Plus a **LOUD boot-log warning** on the harden-on-absent branch (diagnosability). ⚠️ Guard in review: do NOT let a cleanup revert the source to the env check — same code, opposite default, re-opens the hole.
- **Daemon (B):** 3-STATE via `classifySession` (the helper already built for #32's resume guard, `src/session-record.js`), NOT 2-state `isAdversarySession`. record=adversary→cage+flag; record=normal→trust; record=ABSENT/unreadable→**LOUD fail-closed**: refuse to seat as a trusted-normal peer AND do not silently brand adversary; surface `[unverifiable session — security record missing]` to the human. 3-state (not the firewall's 2-state) because the daemon's mislabel has teeth — silently branding a legit absent-record session (pre-#32 legacy, a human-wiped record, an un-run backfill) as adversary is a *new* availability bug.

## Live test + A/#40 refutation — and the option-1 SNI-pinning proxy (BUILT, pending live verify)

**A/#40 was refuted in the live test**, with a hard receipt. A caged Pierre ran
`curl -k --resolve example.com:443:160.79.104.10 https://example.com/` → a **full TLS handshake completed** to the pinned Anthropic IP, and the edge served a cert for **CN=example.com** (`server: cloudflare`, a `cf-ray`). Controls: a foreign SNI at a NON-pinned IP (8.8.8.8) died at L3/L4 (`rc=7` → the firewall blocks by IP, and there is NO local TLS-inspecting proxy); a second foreign host at the pinned IP rode identically. So `160.79.104.10` (api.anthropic.com) is a **shared Cloudflare edge** — the cage pins the IP but not the SNI, the iptables/ipset firewall is L7-blind, and the old boot-check was a false-green (a 4xx with `rc=0` is the foreign origin ANSWERING, not a refusal — the same probe-1 trap, re-shipped). **The cage is network-restricted but NOT exfil-sealed:** a malicious/injected adversary can `--resolve attacker-cf-zone:443:<pinned-ip>` and POST the repo out to an attacker-owned Cloudflare zone.

**Option 2 (LANDED this commit):** the boot probe is now diagnostic-only + rc-based — it logs a LOUD "cage is NOT exfil-sealed" WARNING (does NOT abort), and the docs + the launch banner stop claiming a seal. Feature stays usable; the hole is documented, not hidden.

**Option 1 (BUILT 2026-06-20 — staged, pending Pierre diff-review + the LIVE verify-gate): a host-side SNI-pinning egress proxy.** The real seal. Needs an image rebuild (init-firewall.sh changed).

As built:
- **`src/proxies/sni-proxy.js`** (NEW) — a host CONNECT proxy beside clipboard/notify (127.0.0.1:`<port>`, reached via `host.docker.internal`). Accepts `CONNECT host:443`, replies `200`, then **peeks the in-tunnel ClientHello SNI** and forwards only if SNI ∈ {`api.anthropic.com`, `platform.claude.com`} — connecting out to the **validated SNI host, not the CONNECT host**, so a foreign SNI smuggled inside `CONNECT api.anthropic.com` is caught. Peek-and-tunnel, **NO MITM/cert**. Bounded parse (≤16 KB ClientHello), 15 s anti-slowloris peek deadline (cleared once the tunnel is live), drops + logs anything else.
- **`mrc.js`** (cagedAdversary only) — allocates a port, starts the proxy, pushes `HTTPS_PROXY`/`https_proxy` → `http://host.docker.internal:<port>` + `NO_PROXY=localhost,127.0.0.1,host.docker.internal` + `MRC_SNI_PROXY_PORT`. **Fails the launch closed** if the proxy can't start (no proxy ⇒ no sealed path out). Normal sessions: no proxy, no env, direct egress unchanged.
- **`init-firewall.sh`** (cage branch only) — the cage's direct allowlist is now **empty** (`ALLOWED_DOMAINS=()`) ⇒ the (empty) ipset matches nothing ⇒ **zero direct external egress**; `MRC_SNI_PROXY_PORT` added to `HOST_PORTS`. Boot self-check flipped from the A/#40 *warning* to a real fail-closed seal: (a) `1.1.1.1:443` direct MUST be blocked (rc 7/28) else abort; (b) a foreign SNI **through the proxy** (`curl -x proxy https://example.com`) MUST be refused (rc≠0) else abort — *this is the A/#40 fix asserted at boot*; (c) `api.anthropic.com` **through the proxy** reachable (liveness warning, not abort).
- **`test/sni-proxy.test.mjs`** (NEW, 24/0) — SNI-parser fixtures + loopback e2e: allow tunnels bytes, foreign SNI dropped, **the smuggle** (`CONNECT api.anthropic.com` + foreign SNI inside → dropped, nothing exfiltrated), split-read, case-insensitive, CONNECT framing.
- ⚠ **Load-bearing assumption:** Claude Code routes its API calls through `HTTPS_PROXY` (a documented enterprise feature). The live model call is the proof; if it somehow ignored the env, the fallback is a transparent iptables-`REDIRECT` proxy (heavier — not expected to be needed).

**Verify (the LIVE gate — the real proof; A/#40 was only ever caught live):**
1. A caged Pierre runs the original ride `curl -k --resolve example.com:443:<pinned-ip> https://example.com/` (direct) → must die at L3/L4 (rc 7) now that direct egress is gone.
2. `curl -x http://host.docker.internal:<MRC_SNI_PROXY_PORT> https://example.com` (foreign SNI **through** the proxy) → refused (rc≠0), NOT a 403/200.
3. The **real agent** must actually talk to the model (statusline rate gauges populate, it responds) — this, NOT the `curl -x <proxy> https://api.anthropic.com` smoke test, is the proof `HTTPS_PROXY` is honored. Pierre F-round-2: curl's ClientHello ≠ the agent's, so the curl can pass while the agent's hello is dropped. **Watch the proxy's `sni-proxy` DROP log: a line `DROP non-canonical/unparseable ClientHello via CONNECT api.anthropic.com` means the strict parser refused the real agent's hello (the over-strict brick) → relax to reassemble-then-strict-parse.** A clean log + a responding agent = the seal holds without bricking.
4. A NORMAL session: web (`--web`), clipboard, and model all still work — the proxy is adversary-cage-only.

**Pierre's two-round verdict (2026-06-20):** F1/F2/F4 closed by construction (he threw fragmentation/trailing/dup-ext/dup-host_name/non-LDH/punycode/IDN at the strict parser — all DROP; confirmed the dup-detection is real, not an early-return). F3 banked. F5 closed in staged + the trap comment. The one residual he raised: the strict parser is intentionally STRICTER than BoringSSL (single-record only), so the risk flips from exfil to fail-closed **bricking** if a real agent ever fragments its hello — kept single-record-strict on purpose (reassembly is differential-risk surface on a security boundary; Node doesn't fragment sub-16KB hellos), made it observable via the DROP log, with the reassemble-then-strict-parse relax-path documented in `sni-proxy.js`. His ship gate: run step 3 with the **real agent**, DROP log clean → ship.

> **⚠ STANDING OBLIGATION (Pierre's final point — re-run the live gate on EVERY rebuild / agent bump, not once).** This seal is load-bearing on a binary we don't control and that updates underneath us: Claude Code must (1) honor `HTTPS_PROXY` and (2) keep its ClientHello inside the strict-canonical subset. Crucially, **both couplings fail CLOSED** — an agent that bypasses the proxy hits the empty ipset (bricked), and an agent whose hello goes non-canonical hits a strict-parse drop (bricked); neither leaks. So *security* is robust across agent updates (the firewall backstop doesn't care what the agent does), but *functionality* is coupled to the agent's TLS/proxy internals — and the image disables auto-update and pulls a **fresh** Claude Code on every rebuild. A one-time gate proves one version; nothing in the tests re-checks the next. So the 4-step live gate above is a **recurring obligation tied to every image rebuild / agent bump**, not a one-shot — else we eventually ship a Pierre that launches dead with a `DROP non-canonical… via CONNECT api.anthropic.com` nobody is watching for.
